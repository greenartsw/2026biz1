# 프로젝트 작업 및 Git 백업 규칙

이 저장소의 기준 원격 저장소는 `https://github.com/greenartsw/2026biz1.git`, 기준 브랜치는 `main`이다.

## 필수 원칙

- 작업 시작 전에 현재 저장소, 브랜치, 변경 상태와 원격 연결을 확인한다.
- 사용자 변경을 덮어쓰거나 버리지 않는다. 예상하지 못한 로컬 변경이 있으면 먼저 범위를 확인한다.
- 원격 변경을 확인하지 않은 상태에서 푸시하지 않는다.
- 기능 변경은 관련 검사를 통과한 뒤 커밋한다.
- 사용자가 작업 완료와 GitHub 백업을 요청한 경우 커밋만 하고 끝내지 말고 푸시까지 수행한다.
- 푸시 후 로컬 `HEAD`와 `origin/main`의 커밋 SHA가 같은지 확인한다.
- SHA가 다르거나 작업 트리가 깨끗하지 않으면 백업 완료라고 보고하지 않는다.
- 강제 푸시, 기록 재작성, `git reset --hard`, 사용자 파일 삭제는 명시적 승인 없이 수행하지 않는다.
- 비밀번호, API 키, 토큰, 개인정보가 포함된 파일은 공개 저장소에 커밋하지 않는다.

## 작업 시작 절차

1. `docs/NEXT_SESSION.md`와 `.agents/skills/sync-project-backup/SKILL.md`를 읽는다.
2. `git status -sb`, `git remote -v`, 현재 브랜치를 확인한다.
3. `git fetch origin --prune`으로 원격 상태를 갱신한다.
4. 로컬 변경이 없고 `main`이 원격보다 뒤처졌으면 fast-forward 방식으로만 갱신한다.
5. 로컬 변경이나 분기 상태가 있으면 자동 병합하지 말고 먼저 보존 및 충돌 가능성을 판단한다.

## 작업 종료 및 백업 절차

1. 변경 파일과 diff를 검토해 작업 범위만 스테이징한다.
2. 관련 문법 검사나 실행 검사를 수행한다.
3. 의미가 드러나는 메시지로 커밋한다.
4. 원격 최신 상태를 다시 가져와 비선형 변경 여부를 확인한다.
5. 사용자의 완료 요청 범위에 따라 현재 브랜치를 푸시한다. 이 저장소의 일상 백업은 `main` 동기화를 기준으로 한다.
6. `.agents/skills/sync-project-backup/scripts/check-sync.ps1`을 실행한다.
7. 최종 보고에 브랜치, 커밋 SHA, 푸시 결과, 검사 결과를 포함한다.

## 파일 기준

- `C:\Users\USER\Downloads\2026biz1-main`은 ZIP 추출본이며 Git 저장소가 아니다. 참고나 비교 용도로만 사용한다.
- 실제 작업과 백업은 이 `2026biz1` Git 저장소에서만 수행한다.
- 프로젝트2는 프로젝트1 운영본을 직접 덮어쓰지 않고 별도 브랜치 또는 하위 폴더에서 재사용한다.
